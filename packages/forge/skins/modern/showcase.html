How the SmartClient skin actually works

Short version of what I see in the skin_styles.css you uploaded, and what's realistically modernizable.

The skin = a CSS classname contract

SmartClient widgets render real DOM with hard-coded class names. Your skin styles those classes. The framework owns the structure; your CSS owns the look.

  • .button, .buttonOver, .buttonDown, .buttonSelected, .buttonDisabled, …
  • .windowHeader, .windowBody, .dialogBody
  • .sectionHeader, .sectionItem
  • .formCell, .formTitle, .textItem, .selectItem, .checkboxItem
  • .tabTitleSelected, .tab, .tabBar
  • .listGridCell, .listGridSelectedCell, .gridHeader

Your file = compiled SCSS output

The header literally says "DO NOT edit this .css file directly". The truth lives in skins/EnterpriseSeries/sass/_ThemeGraphite.scss + _base.scss. Two real paths to modernize:

  1. Override layer. Ship a second stylesheet loaded after skin_styles.css that re-paints the same classnames. No SCSS, no SC code change. Wins ~90% of the surface.
  2. Fork the SCSS. Copy the EnterpriseSeries sass to a new theme (_ThemeModern.scss), remap variables, regenerate. Cleaner, but you live inside their conventions.

What's easy

  • Color, type, spacing, radii — pure CSS variables on a wrapper class.
  • Buttons, ToolStrip, SectionStack, TabSet, DynamicForm, Window/Dialog headers.
  • ListGrid rows, hover, selection, headers.
  • Light + dark on the same skin (toggle a class on .sc).

What's annoying

  • Sprite-image dependencies — scrollbar grips, sort/group arrows, twisties — referenced via background-position. Either replace the sprite, or hide them and re-draw in CSS (mask + currentColor, like the icons in this prototype).
  • Base64 SVG gradients baked into properties — harmless, just verbose; an override stylesheet replaces them.
  • Anything sized in pixels by the framework (fixed row heights, tab corner images). Usually fine, occasionally needs height/border-radius overrides.

Recommended path

Ship a Modern skin as an override stylesheet first — load it after skin_styles.css, gate it on a <body class="skin-modern">, and offer light/dark via a second class. Spend zero SCSS time. Once it's stabilized and you have signoff on the visual direction, fork the SCSS for proper distribution. The CSS you see in this showcase (the .sc.skin-modern rules) is roughly what that override stylesheet looks like in practice — distilled from your skin_styles.css, re-targeted to the same class contract.
