Can CSSZ replace SmartClient’s skin system?

CSSZ is an HTML-driven, recursive, variable-only design system. SmartClient’s skin system is a class-name contract plus image registry plus JS bootstrap. They overlap on tokens and conflict on everything else. Here’s where each one wins, where they collide, and how to use them together.

Source: SmartClient Skinning docs Source: SVG Symbols overview Source: Sass Skinning guide Source: cssz README + cssz.css/.js Reference: Modern override skin (this project)

TL;DR

CSSZ is an excellent token + theme-scoping layer for SmartClient — and a poor replacement for its skin system. Use CSSZ to drive the colors, type, spacing, radii, shadows, density; keep SmartClient’s class-name contract and load_skin.js for everything else (image registry, AutoChild defaults, sprite metrics, suffix-based statefulness).

The natural fit is a three-layer stack: SmartClient’s base CSS owns structure & statefulness, a thin SC↔CSSZ adapter stylesheet rewrites SC’s hard-coded colors/fonts/sizes to var(--cssz-*) with safe fallbacks, and CSSZ blocks on any container provide the live, recursive theme. You get HTML-driven theming, sub-tree scoping, and instant dark-mode without abandoning the SmartClient runtime.

01How SmartClient skins actually work

From the official skinning docs, a SmartClient skin is not just a stylesheet — it’s a coordinated bundle of four moving parts:

SmartClient skin = bundle
├─ skin_styles.css → class-name contract: .button, .buttonOver, .listGridCell, .windowHeader, ...
├─ images/ → tree of PNG/SVG sprites organized by component
├─ load_skin.js → runtime defaults: AutoChild srcs, capSizes, baseStyles, useCSS3
└─ template/sass/ (modern skins only) → _base.scss + _userSettings.scss → compiled .css

Three things matter for an alternative system to be a real replacement:

(a) Stateful suffixing is part of the contract

SmartClient mutates DOM class names and image URLs by appending state suffixes — buttonOver, buttonDown, buttonSelected, buttonDisabled, listGridCellSelectedOver, etc. The framework reads style declarations to extract size/font info and reportedly fails on multi-class selectors and at-rules: only single class-name selectors are officially supported. That means any custom system has to expose those exact class names with the right metrics, or the framework misbehaves.

(b) load_skin.js isn’t optional

It declares image URLs ("[SKIN]/Window/minimize.png"), ImgTab cap sizes, EdgedCanvas segments, sprite metrics, and the useCSS3 flag. It also wires AutoChild defaults via changeDefaults(). None of this can be moved to CSS variables — it’s consumed by JS at component-creation time.

(c) Sass templates already do CSS variables — sort of

Modern Flat-series skins (Tahoe, Stratus, Twilight, Obsidian) ship a _base.scss with named variables ($highlight_color, $standard_button_border_radius, …) and a _userSettings.scss for overrides. The compile step bakes them into the static skin_styles.css. This is design-time tokenization — not runtime, not recursive, not HTML-driven.

(d) Density and SVG symbols are JS-side

Density is JS: resizeFonts(n) + resizeControls(n) after skin load, before any component creation. SVG symbols use the sprite:svg:#id URL format with inline modifiers (color, size, cssClass, statefulId). Both bypass CSS entirely.

02CSSZ in one page

CSSZ is three things stacked on top of CSS Custom Properties:

  1. A naming convention. Every variable is --cssz-*. Engine classes are .cssz-*.
  2. An engine stylesheet (cssz.css) of ~50 lines: each .cssz-{component} rule reads variables with fallbacks and applies no concrete tokens of its own.
  3. A scoping discipline. The theme lives in inline style on a container; nested containers can override any subset; everything else inherits via CSS’s normal cascade.

The optional cssz.js is 24 lines: it just turns { key: value } into setProperty('--cssz-key', value) on a target element. No dependency, no build step, no diff engine.

What CSSZ is good at

  • Recursive scoping — a sub-tree can override --cssz-bg and everything underneath updates without selectors.
  • HTML as source of truth — the theme of a region is visible right next to its markup.
  • Trivial dark mode and tenant theming — flip a few variables on <html> or any container.
  • Zero build pipeline — compared to Sass compile + cache-bust, this is just a stylesheet load.
  • Composable with other CSS — variables don’t care who reads them. Any third-party stylesheet with var(--cssz-*) will participate.

What CSSZ doesn’t do

  • Doesn’t describe state machines (no equivalent of .buttonOver, no rollover/selected logic).
  • Doesn’t register images, sprites, or AutoChild srcs.
  • Doesn’t affect JS-side things — Density, ImgTab cap sizes, useCSS3.
  • Doesn’t solve sprite metric coupling (background-position offsets that break when row heights change).

03Fit analysis: where they overlap, where they collide

SmartClient owns

  • Component DOM structure
  • Class-name contract (.buttonOver, .listGridCellSelected, …)
  • State machine + suffix mutation
  • Image registry & sprite URLs
  • load_skin.js AutoChild defaults
  • Density (resizeFonts/resizeControls)
  • SVG symbol pipeline (sprite:svg:)

CSSZ owns (when adopted)

  • Token vocabulary (color, type, space, radius, shadow)
  • Theme scoping (sub-tree overrides)
  • Runtime theme switching (no recompile)
  • Tenant / per-page / per-widget theming
  • Designer-authored theme blocks (HTML-pasteable)
  • Reactive themes (time, battery, status)

The collision is at the value layer of skin_styles.css: SmartClient’s class rules contain hard-coded color: #29333c, background: linear-gradient(...), font: 11px Arial. CSSZ wants those values to be var(--cssz-*). That collision is small and surgical to fix — and that’s exactly what the “modern override skin” we already shipped does, just under SC’s naming instead of --cssz-*.

Three replacement strategies, ranked

1. Total replacement

don’t

Drop SmartClient skin, ship only CSSZ engine + classes on SC’s DOM. Breaks state suffixing, image registry, AutoChild defaults, ImgTab metrics. Months of regression work; permanent fork from upstream.

2. Override layer (current plan)

ship now

SmartClient skin loads first; a second stylesheet retargets the same class-name contract. No cssz vocabulary. This is what your handoff/skin-modern.css already is.

3. Adapter + CSSZ tokens

strategic

Same as #2, but the override stylesheet’s values are var(--cssz-*) with the SC defaults as fallbacks. Now any container can rescope the theme. Costs ~1 day on top of #2.

05The hybrid: SmartClient + CSSZ adapter

This is the architecture I’d ship. Three layers, very thin glue.

Page boot order
1. load_skin.js → images, AutoChild defaults, density, useCSS3 — unchanged
2. skin_styles.css → SmartClient’s class contract + state suffixes — unchanged
3. cssz.css → optional: any standalone CSSZ engine classes you want for non-SC HTML
4. sc-cssz-adapter.css → retargets SC classes to var(--cssz-*) with SC defaults as fallbacks
5. theme block(s) in HTML → on <body>, on any panel, on any subtree — recursive

The adapter is “modern override” with one tweak

You’ve already built it once. Compare:

Before — values baked into class

/* skin-modern.css (what we shipped) */
.button {
  background: var(--sc-button-bg);
  color: var(--sc-text);
  border: 1px solid var(--sc-border);
  border-radius: var(--sc-radius);
}

After — adapter speaks CSSZ

/* sc-cssz-adapter.css */
.button {
  background: var(--cssz-button-bg, var(--cssz-surface, #fff));
  color: var(--cssz-text, #000);
  border: 1px solid var(--cssz-border, #ababab);
  border-radius: var(--cssz-radius, 0);
}

Now any SC component reads from CSSZ scope. Drop a CSSZ block on <body> and the whole app re-themes; drop one on a panel and only that panel does. The single-class selector rule is preserved. SC’s state machine still mutates classes, and each state still resolves through CSSZ vars.

State suffixes still work

.buttonOver     { background: var(--cssz-button-bg-over,  var(--cssz-button-bg)); }
.buttonDown     { background: var(--cssz-button-bg-down,  var(--cssz-button-bg)); }
.buttonDisabled { background: var(--cssz-button-bg-disabled, var(--cssz-surface-mute));
                  color:      var(--cssz-text-mute); }

SVG symbols become themable for free

SC’s SVG symbols inherit color via currentColor. Set color: var(--cssz-icon, currentColor) on SC’s icon base styles in the adapter, and every sprite:svg: graphic that uses currentColor picks up your CSSZ scope. Per-tenant icon coloring with zero asset swap.

06Concrete example: tenant + dark mode in one HTML page

<!-- one global theme -->
<body style="
  --cssz-bg: oklch(98.5% .002 250);
  --cssz-text: oklch(20% .01 250);
  --cssz-border: oklch(91% .005 250);
  --cssz-button-bg: #fff;
  --cssz-button-bg-over: oklch(96% .005 250);
  --cssz-radius: 6px;
  --cssz-font: 'Inter', system-ui, sans-serif;
">

  <!-- standard SC content gets the global theme -->
  <div class="sectionStack">...</div>

  <!-- one panel rescopes to dark on the fly -->
  <div style="
    --cssz-bg: oklch(18% .005 250);
    --cssz-text: oklch(96% .003 250);
    --cssz-border: oklch(28% .008 250);
    --cssz-button-bg: oklch(22% .008 250);
  ">
    <div class="button">Save</div>   <!-- dark -->
    <div class="buttonOver">Hovered</div>
  </div>

  <!-- a third panel pulls in “Acme tenant” brand -->
  <div style="
    --cssz-button-bg: oklch(58% .15 25);
    --cssz-button-text: #fff;
    --cssz-icon: oklch(58% .15 25);
  ">...</div>
</body>

Three themes coexist on the same page. SmartClient’s components don’t know any of this is happening. They render their normal class names; the adapter resolves them against whichever CSSZ scope is closest.

07Risks & gaps you should know about

The known-hostile bits of SC’s skin contract

  • Sprite metrics. Some skins (Enterprise, EnterpriseBlue, Graphite) embed background-position offsets that break if you change row height or padding via CSSZ. Either keep SC’s baked metrics or set isc_spriting = "off".
  • Single-class selectors only. The framework reads style declarations from CSSOM and reportedly doesn’t handle multi-class selectors or @-rules well. Your adapter must stay flat.
  • Sizes that SC reads from CSS. Things like row height come back to JS via getComputedStyle. Changing them through --cssz-row-h works visually but make sure SC’s autosize logic still gets a number — don’t set heights via calc() that produces a value the framework can’t round-trip.
  • load_skin.js sets fixed pixel sizes for capSize, AutoChild srcs, etc. These don’t flex with CSSZ and remain JS-controlled.
  • Density still has to be JS. If you want CSSZ-driven density, expose --cssz-density and have a small bootstrap script call resizeFonts/resizeControls from it before component creation.

Things from the cssz research doc to be skeptical of

  • “Transpile Bootstrap to CSSZ” — fine for Bootstrap; SmartClient is too dynamic. Don’t try.
  • “Self-theming HTMZ fragments” — beautiful pattern, but SC components are created by JS, not loaded as HTML fragments. The CSSZ block on the host element still works, but the fragment-loading story doesn’t apply.
  • “CSSZ Crawler / AI generative themes” — orthogonal to SmartClient. Cool, but not part of this evaluation.

08Recommendation

Adopt CSSZ as a tokenization + theme-scoping layer over SmartClient. Don’t replace the skin system.

Phased plan

  1. Ship the modern override skin you already have as skin-modern.css. Validates the visual direction with zero architectural risk. today
  2. Refactor it into sc-cssz-adapter.css — same file, just rename internal vars from --sc-* to --cssz-* and add SC defaults as fallbacks. ~1 day of mechanical edit. week 1
  3. Add a default CSSZ block on <body> (light) and a class-toggled dark variant. Verify dark/light parity in the showcase. week 1
  4. Document the CSSZ vocabulary for designers (the same subset you used for the override: bg, surface, text, mute, border, accent, focus, radius, font, density-scale, plus per-state variants for buttons/rows). week 2
  5. Pilot per-tenant theming by injecting a CSSZ block at login. Prove that the SC components reflect the brand without recompile. milestone
  6. Defer SVG symbol theming & density-via-cssz until the adapter is stable. Both are additive.

What this gives you that pure SC skinning doesn’t

  • Live dark mode (no Twilight/Obsidian recompile).
  • Per-tenant brand without per-tenant build artifacts.
  • Sub-tree theming — wrap any panel, override the vars, done.
  • HTML-driven theme blocks — designers paste, devs ship.
  • CSSZ outside SC: any non-SC HTML on the page (marketing zones, embedded charts) participates in the same tokens.

What it costs

  • One small adapter stylesheet you maintain across SC upgrades.
  • A naming-convention discipline: every visible SC class style reads only from var(--cssz-*) with a SC-default fallback.
  • Awareness of the “hostile bits” in §7 — sprite metrics, pixel measurements SC reads back, density.

Bottom line: CSSZ doesn’t replace SmartClient’s skin system, it completes it. SC owns structure and statefulness; CSSZ owns design tokens and runtime scoping. The combined system is smaller than Tahoe’s Sass build and more flexible than any builtin skin.
